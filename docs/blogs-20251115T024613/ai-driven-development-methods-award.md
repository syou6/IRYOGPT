Devin, Claude Code, Codex … 生成AIの進歩はあまりにも早い。

AIアシスタントは多くの会社から毎日のようにリリースされ、それぞれ得意分野が異なり、日々アップデートされ続けている。数週間前まで最先端だった手法が、翌週にはすでに時代遅れになっている、非常に流れの早い分野だ。

正直、「今どれを使えば一番メタ（＝本質的に効率的）なのか」が分からなくなる瞬間がある。

だからこそ私は、実際のアプリ開発でAIを“チームメンバー”として扱い、役割を明確に分担させる方法を試した。結果として、アメリカの国際アプリ大会「Shipathon 2025」で、私の開発したアプリが入賞を果たすことができた。

今回は、私がそのとき採用した「AI駆動開発メソッド（AI-Driven Development）」の実践手法を紹介したい。



# 1. 以前はClaude Codeが主軸だった

開発初期、私の主戦力は Claude Code だった。Claude Codeは、自然言語理解が極めて優れており、長いコンテキストを把握したうえでコードを生成してくれる。「ここをリファクタして」「型安全にしたい」など、ふんわりした指示でも理解して形にしてくれる柔軟さがあった。

ただし、実際に使い続ける中で、Claude Codeには明確な得意・不得意があることも分かってきた。たとえば「コードを書く」こと自体は得意だが、「どこにpushするか」「どんなコミットメッセージを付けるか」といった開発実務のルーチン処理には非常に強い。逆に、アルゴリズム設計やUIデザインなど、エンジニアリングの深い層になると苦手なことが多い。

私はその特性を逆手に取って、Claude Codeを“Push担当”にした。具体的には、次のような流れだ。

私がローカルでコードを大きく変更する。

Claude Codeに「いろいろ修正したから、pushして」と伝える。

Claude Codeが現在の変更内容を解析し、コミットすべきファイルとその理由を整理してくれる。

変更点の要約を自動でコミットメッセージとして生成し、「この内容でpushしていい？」と聞いてくる。

私は「Yes」またはEnterを押すだけ。

この仕組みを導入してから、地味に面倒だった「コミットメッセージ地獄」から解放された。

特に中・大規模のプロジェクトでは、後からバグの原因を追跡するために、コミットメッセージの品質が非常に重要になる。Claude Codeは自然言語処理が得意なので、差分の意味を的確に要約し、「何を」「なぜ」変えたのかを日本語または英語で分かりやすく書いてくれる。「AIがgit pushまでやる時代が来た」と感じた瞬間だった。

いや、以前からできていたけれど、Claude Codeの取り柄はこれだけが残った、というのが正しいのかもしれない。

# 現在の主軸はCodexへ

ただし、最近の主戦力は完全に Codex（GPT-based coding engine） に移行している。

理由はシンプルで、コーディング能力が圧倒的に高い からだ。Codexは関数単位・モジュール単位の生成精度が非常に高く、構造的なリファクタやアーキテクチャ設計にも強い。Clean Architectureの層分け、Repository/UseCaseの依存関係、SwiftDataのエンティティ設計など、かなり複雑な構造でも迷わず正しいコードを出してくれる。

たとえば、私のアプリ「メモるん」では、

Presentation層（SwiftUI）

Domain層（UseCase/Entity）

Data層（Repository実装）

Infra層（ファイルシステム、APIアクセス）という4層構造を採用している。

このような構造では、単一ファイルだけではなく「層間の責務」を意識したリファクタが必要になる。ここでCodexの真価が発揮される。「このエンティティをUseCaseに統合して」「RepositoryImplをData層に移して依存逆転を保って」など、自然言語での設計変更指示を的確に実装へ落とし込んでくれる。

これは、Claude Codeでは非常に難しかった。Codexでは、最初からクリーンアーキテクチャで作ることは容易いけれど、Claude Codeではかなり丁寧に指示をして、できるかどうかだった。そのため、私は最初クリーンアーキテクチャで作らなかったため、今、技術負債の解消のため、ずっとこれをやっている。

# それでも解決できない場合はOOで解決する！

一方で、Codexにも苦手分野がある。それは「問題の本質を抽象化して説明すること」だ。

Codexは（Claude Codeよりもマシとはいえ）、全体のコードを読み、全体的になんとなくうまく動く、程度のコードを書くことは非常に上手い。しかし、デザインや各層の役割分担を明確に書くのは、そこまで上手ではない。

そこで私は、ChatGPT（GUI版）を“解析担当”として使うようにしている。

ふざけてるのかと思うかもしれないけれど、これは事実だ。Codexよりも、ChatGPTにプロンプトとして直接投げた方が、遥かに精度が高い回答が帰って来る。おそらく、Codexは関係のないコードをたくさん読んでしまうことでコンテキストを消費してしまい、無駄な思考を巡らしてしまうからだろう。

ChatGPTの使い方は単純で、Codexが行き詰まった仕様を実現するために現在の壊れたデザインの写真を投げて、「写真とメッセージバブルの隙間の原因を教えて」などと指示をする。

ChatGPTは、実行環境を持たない代わりに、論理的な説明力が圧倒的に高い。それによって「原因」と「解決方針」が明確になったら、私はただ、その見解を見て、正しそうならその知見をCodexに戻す。ダメそうなら指摘をする。失敗したら、Claude Codeに「 feature/*の最新にフォールバックして。今のプログラムは消していい」と書いて、もう一回やり直すだけだ。

# AIの使い方で、成果物の結果が変わる！

多くの人が、AIを“ツール”として扱っている。しかし、私は、AIを"私の分身"として扱う。

その違いは単純だが、決定的だ。

AIは単なるツールではない。私の代わりになってやってくれる、業務委託の社員だ。

小規模のプロジェクトにおいて、開発を並列に行うのはとても難しい。基本的には直列に行うしかない。PERT図を作り並列にできる仕事を分析して複数のパソコンでやるのは流石に難しい。

だから、できそうな仕事を作り、それを投げ、AIが作業している間に指示文を作成したり休息を取ったりし、作業が終わったらすぐに検収をして次の指示を渡す。生成AIは月契約だから、できる限り働いてもらう。そんな使い方が現在、主要になりつつある。

開発という行為が「人間が書く」から「AIと書く」へと変わりつつある今、重要なのは「どのAIをどう組み合わせるか」を設計する力だ。AIを1つのツールとしてではなく、複数の人格を持つチームとして統合する力だ。

このメソッドを通じて、私は改めて確信した。

未来の開発者は、コードを書く人ではなく、AIを指揮するディレクターになる。

そして、その第一歩は、仕様や思想を正確に言語化するところから始まるだろう。
