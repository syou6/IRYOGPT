# プレミアムSaaS UI改善ロードマップ

> `docs/premium-saas-ui-guidelines.md` を実際の画面へ落とし込むための作業段階を定義したドキュメント。各フェーズでの目的・主なインプット・アウトプット・完了条件を明確化し、デザイナー／開発者／AI協業をスムーズにすることを狙う。

---

## フェーズ0: キックオフ & 成功指標の合意しよや
- **目的**: ブランドトーン（緑×黒のプレミアム感）とビジネスKPIを共有し、改善対象画面と優先順位を決定。
- **インプット**: 既存プロダクト画面キャプチャ、UX課題リスト、ビジネス目標。
- **アウトプット**: 対象画面リスト、成功指標（例: トライアル→有料転換率5%向上）。
- **完了条件**: 関係者が合意した要求仕様書／Miroボードが存在。

## フェーズ1: UI監査 & ギャップ分析
- **目的**: 現状UIをガイドライン（HIG, Material, Fluent 等）と照合し、差分を可視化。
- **インプット**: フェーズ0の資料、`premium-saas-ui-guidelines.md`。
- **アウトプット**: 画面別スコアカード、問題カテゴリ（配色/余白/階層/アクセシビリティ）。
- **完了条件**: 優先度付きバックログ化（Jira/Linear など）。

## フェーズ2: デザイントークン & システム定義
- **目的**: カラー・タイポ・スペーシング・Elevation・モーションをトークン化し、Figma/コード双方で使える形に。
- **インプット**: 既存スタイルライブラリ、カラーパレット案、Type scale。
- **アウトプット**: Figma の Styles/Variables, JSON/TSベースの Design Tokens, 変更ログ。
- **完了条件**: トークン変更のみで主要UIの80%を制御できる状態。

## フェーズ3: コアコンポーネント再設計
- **目的**: ボタン・フォーム・カード・モーダル等の共通コンポーネントをプレミアム仕様へ刷新。
- **インプット**: トークン、既存コンポーネントリスト、使用頻度データ。
- **アウトプット**: 新UIキット（Figma コンポーネント＋コードスニペット）、ステート仕様書。
- **完了条件**: 主要コンポーネントの 90% が新トークン対応。ドキュメント化済み。

## フェーズ4: 主要画面リデザイン（Marketing / Auth / Dashboard / Plans）
- **目的**: 画面単位で情報構造・CTA導線を再配置。Stripe完了後の「運営が学習」といったUXメッセージも組み込む。
- **インプット**: UIキット、ユーザーフロー、計測データ。
- **アウトプット**: 各画面の hi-fi モック＋インタラクション仕様、コピー草案。
- **完了条件**: ステークホルダー承認済みの hi-fi / プロトタイプが揃う。

## フェーズ5: マイクロインタラクション & 状態設計
- **目的**: ホバー／フォーカス／ローディング／エラー／サクセスなどの状態を統一し、ブランド体験を仕上げる。
- **インプット**: コンポーネント仕様、アクセスログ（どこで離脱するか）。
- **アウトプット**: Motion spec（200–300ms, Easing）、Lottie/GIF試作、アクセシビリティノート。
- **完了条件**: UI Kit に各状態が紐づき、実装チームが迷わない状態。

## フェーズ6: 実装同期 & デザインレビュー
- **目的**: フロント実装チームと密連携し、トークンやコンポーネントが Next.js / Tailwind / Supabase へ正しく反映されるようガイド。
- **インプット**: フェーズ2〜5の成果物、技術スタック情報。
- **アウトプット**: 実装計画、Storybook/Chromatic等での差分チェック体制。
- **完了条件**: 主要画面のデザインQAパス、アクセシビリティ検証（Lighthouse, axe）済み。

## フェーズ7: 計測 & 継続改善
- **目的**: リデザイン効果を指標で追跡し、追加の仮説検証サイクルを回す。
- **インプット**: リリース後のユーザー行動データ、定性ヒアリング。
- **アウトプット**: KPIレポート、次イテレーションの改善計画。
- **完了条件**: 目標KPIへの影響が定量的に把握でき、次フェーズに反映されている。

### 次アクション詳細

#### 1. UI監査テンプレート整備（Phase 1）
- 監査シート列構成: `画面ID / 問題カテゴリ / 優先度 / ガイドライン節 / Evidence / 推奨施策`。
- `ガイドライン節` には `premium-saas-ui-guidelines.md` の章番号（例: `6. ナビゲーション`）を記入し、優先度付けの根拠を明示。
- Jira/Linear へ移す際も同じ番号をタグ化しておくと Phase 4 での確認が容易。

##### Notion/FigJam 複製用テンプレ
| 画面ID | 問題カテゴリ | 優先度 (H/M/L) | ガイドライン節 | Evidence（スクショ or FigJamリンク） | 推奨施策 / Owner | ステータス |
| --- | --- | --- | --- | --- | --- | --- |
| DASH-Auth-01 | 配色/コントラスト | H | 2. カラーパレット, 8. アクセシビリティ | `figjam.com/...` | CTA背景をPrimary/Deepへ統一（担当: design-lead） | Backlog |
| DASH-KPI-02 | 情報階層 | M | 6. ナビゲーション, 7. インタラクション | `notion.so/...` | KPIカードを3つに制限、二次情報を折りたたみ | In Review |

- Notion用: 上記表をDatabaseとして貼り付け、`ステータス`をSelect、`優先度`をMulti-selectで設定。
- FigJam用: Stickyテンプレを同じ列名で作成し、`ガイドライン節`はドロップダウン。Notion IDと連携する場合は `画面ID` を共通キーにする。

#### 2. トークン案 + アクセシビリティ手順準備（Phase 2）
- 必須アウトプット
  - カラートークン: `Base/Surface/Primary/Accent` をJSON/TS形式で定義し、WCAG AAコントラスト計算結果を記載。
  - タイポトークン: H1–Captionまでの`fontSize/lineHeight/fontWeight`セットと多言語フォールバック。
  - アクセシビリティ検証フロー: `Lighthouse → axe DevTools → 手動Tab移動`の順で実施するチェックリスト。
- Figma Variablesとコードトークンを同時に更新し、差分は`モーション & 状態設計`節へも反映する。

##### トークン草案フォーマット例（Notionコードブロック or Style Dictionary）
```json
{
  "color": {
    "base": { "value": "#040607" },
    "surface": { "value": "#0B1410" },
    "primary": { "value": "#19C37D", "contrastOnBase": 5.8 },
    "primaryDeep": { "value": "#0F8A5F", "contrastOnBase": 4.1 },
    "accent": { "value": "#7AF4C1" }
  },
  "typography": {
    "h1": { "fontSize": 40, "lineHeight": 48, "fontWeight": 600, "font": "Söhne" },
    "body": { "fontSize": 16, "lineHeight": 24, "fontWeight": 400, "font": "Inter" }
  }
}
```

##### アクセシビリティ検証チェックリスト
1. Lighthouse (Desktop, Dark mode) で `Accessibility >= 95` を確保。
2. axe DevTools クイックスキャンで `Serious以上 0` を確認、issueごとに`ガイドライン節`を記録。
3. 手動Tab移動でフォーカスリングが常に視認できるか確認し、`premium-saas-ui-guidelines.md` の 7/8章に反映。
4. カラートークン更新時は contrast-ratio.com で Base×Text, Surface×Text, Primary×Base の比率を再計算しテンプレへ記録。

#### 3. 情報階層レビュー準備（Phase 4）
- Dashboard/Plans画面のワイヤーフレームに「KPIは最大3つ」「Stripe後バナー挿入位置」を注記。
- コピー草案で「運営が学習を進めるので少々お待ちください」を必須文言としてマークし、ローカライズ方針を定義。

### 運用メモ
- 各フェーズ完了時に `docs/premium-saas-ui-guidelines.md` を再確認し、逸脱がないかレビュー。
- Stripe 決済後のバナーや「運営が学習中」等のメッセージは、フェーズ4と5で同時に仕上げて実装反映を忘れない。
- フェーズ2〜4でAI生成を活用する場合は、ガイドラインとこのロードマップを冒頭に読み込ませることで品質を担保する。
